# stock-research skill

> 一个将投研方法论编码为模型约束的 Claude Code skill

---

## 设计哲学

**核心定义**：Skill 的本质是将人的分析方法论编码为模型约束，在方法论关键节点强制执行人的判断标准，在执行过程中给模型合理的发挥空间。

这不是让模型自由发挥，也不是完全规定模型的每一步行为，而是在两者之间找到正确的分界线。

**关键推论**：
- 方法论节点（不编造、不停在表面、横向对比先行）→ 必须显式约束，模型不会自然执行
- 执行过程（如何推理、具体问哪些问题）→ 交给模型判断，过度规定反而制造脆性
- 每次新增规则前的校验标准：这是方法论上的失误（需要约束），还是执行层的失误（接受模型能力边界）？

---

## 三层架构

```
原则层          不编造 / 不停在表面 / 不孤立解读 / 不单向论证 / 可证伪
                ↓ 模型执行（推理空间）
工具层          yfinance（结构化市场数据）/ NotebookLM（公司内部文档）/ Tavily（外部环境）
                ↓ 模型执行（推理空间）
输出标准层      场景化的报告覆盖要求（场景一～六）
```

中间的推理过程是模型的发挥空间，不应被规则化。规则只在三层本身存在。

---

## 能力边界

### 下限（可以保证做到的）

对任何一家公司，从零开始，产出一份不犯低级错误、不遗漏结构性问题的分析框架：财务数据交叉验证、传导机制识别、多空对称、关键未知显式标注。这条下限不依赖用户的行业积累，是skill结构性保证的最低交付质量。

### 上限（最好情况下能做到的）

在用户完全不了解某家公司时，帮助在短时间内建立一个有质量保证的认知起点，并识别出领域内分析师也可能忽略的结构性机制问题。这是skill的天花板——替代"从零开始的第一次深度研究"。

### 适合做什么

- **陌生公司的首次研究**：价值最高，没有替代方案
- **财务数据的机械验证**：交叉核对、识别误导性数字，准确且不知疲倦
- **分析框架的搭建**：给后续深度研究提供骨架，防止遗漏维度
- **无内部覆盖的公司**：团队无人跟踪、无法做渠道调研时，是合理的替代品

### 不适合做什么

- **有资深研究者长期跟踪的公司**：此时skill只是边际工具，核心判断仍然来自人的积累
- **财报前的预判形成**：没有历史跟踪积累，skill无法形成"关键变量应该落在什么区间"的锚定观点，也无法在财报出来后做真正的验证而非重新分析
- **非公开信息的整合**：渠道反馈、行业访谈、市场观察——凡是"听说"类信息，skill无法处理
- **产生投资conviction**：skill能帮助把问题想清楚，但无法替代长期跟踪和真实持仓所产生的判断锐度
- **市场预期差分析**：识别"分析师共识过高导致合理业绩被误读为不及预期"这类判断，skill目前基本缺位

### 一句话定位

**skill是分析起点的质量保障工具，不是分析终点的判断工具。** 它能帮助站对起跑线，但跑到终点靠的是人的行业积累。用它替代"无覆盖公司的第一次研究"，物超所值；用它替代"资深研究者的持续跟踪判断"，注定不够。

## 版本迭代

### v0.51 — 铁律升级（2026-04-24）
**变更**：
- 铁律一：三分类扩展为四分类，新增「关键未知」——对判断有重要影响但无法获取或验证的信息须显式标注，并说明影响方向（偏多/偏空/不确定）。填补了"重要但不知道"的表达盲区。
- 铁律二：从"标准编号不得外露"扩展为"报告是分析文本，不是模板完形填空"——标准和场景规范是质量检查项，不是报告章节结构。

---

### v0.40 — 设计哲学重构（2026-04-24）
**变更**：依据"方法论节点约束 vs 执行过程开放"的设计哲学，对 standards.md 和 scenarios.md 做结构性清理。

**核心改动**：
- 标准二：5层强制序列 → 核心约束 + 参考切入维度（不限于此）+ 输出要求
- 标准三：if-else 分支规定 → 核心约束 + 输出要求
- 标准五：5a/5b 代号 → 政策扫描/已知事件清单；移除时间窗口限制
- 场景六：五维度强制表格 → 参考评估维度（不限于此）
- 新增：全场景共用的提交前自查清单（四条核心方法论检查）
- 全文：清理所有 5a/5b 代号引用

**设计原则落地**：区分"输出须包含什么"（方法论约束）和"如何分析"（模型空间）；参考维度提示替代强制步骤清单。

---

### v0.30 — 铁律一升级（2026-04-24）
**变更**：铁律一从"数字类数据必须来自工具"扩展为"所有实质性陈述必须有来源或显式标注为推断"。

**触发原因**：贵州茅台2025年报分析中出现两个严重事实错误——管理层改革归因错误（将陈华任内落地的改革误归于张德芹）、混淆上市公司总经理与集团党委书记/董事长的权力层级。根本原因是 skill 对定性事实的要求远低于定量数据，允许凭印象推断。

**核心机制**：引入三类强制区分——可查证事实（须有来源）/ 合理推论（须显式标注）/ 假设（须显式标注）。未标注 = 默认为可查证事实 = 必须有可追溯来源。

---

### v0.20 — 分析能力扩展（基于 Opus 诊断分析）
**新增**：
- 商业模式第一性建立（取数前先建立"这家公司靠什么赚钱"的模型）
- 反共识视角（须先锚定共识来源，再说明为何可能错）
- 市场定价隐含假设分析（当前估值背后的假设是否合理）
- 管理层质量与资本配置（5种资本用途历史ROI、内部人持股、薪酬激励）
- 数据交叉验证（净利润与OCF分叉须标注为红旗）

**设计背景**：v0.10 解决了"不编造、有结构、有追因"，但缺少产生非共识洞见的关键机制。

---

### v0.10 — 初始版本
基础框架：数据优先铁律、多层追因、横向对比、多空对称、可证伪框架。

## 待解决的设计问题

- **NotebookLM 集成**：工具尚未安装部署。集成方向已明确：作为公司内部文档（年报/中报/电话会纪要）的全文检索层，补充 yfinance 无法覆盖的子科目明细和附注信息。集成范围限于场景一（完整财报分析），不做全局架构改动。

- **规则最小化**：随着使用积累，持续识别哪些规则是真实失败模式的记录、哪些是防御性过度工程，后者应逐步删除。
